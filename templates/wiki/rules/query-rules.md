# Query Rules

本檔定義「查詢與統整」時的工作方式。回答問題時，優先利用已整理好的 wiki，而不是每次重新回到原始來源做臨時拼裝。目標是讓查詢本身也能反過來強化知識庫。

## 任務邊界

- 本規則只處理「以主題、概念、比較、分析、統整」為主的問題。
- 若使用者的核心要求是某一天、某一段時間讀了什麼，或要整理閱讀紀錄報表，應改走 `wiki/rules/review-rules.md`，不是本規則。
- 就算問題裡出現日期，只要主要目的仍是問主題本身，例如「4 月底讀過的文章裡怎麼解釋 durable object？」核心仍是 query，不是 review。
- 本規則的回答重點是知識結論與依據，不是時間流水帳。
- 不可因為查詢結果剛好引用到某天或某段時間的內容，就把回答退化成閱讀清單或標題羅列。

## 任務目標

當使用者針對知識庫提問，或要求比較、統整、分析多個主題時，你要：

1. 先從 `wiki/index.md` 找相關頁面。
2. 再讀取最相關的 `wiki/concepts/` 與 `wiki/summaries/`。
3. 基於已存在的 wiki 內容做綜合回答。
4. 清楚指出依據來源與不確定之處。
5. 若這次查詢產生高價值的新綜合內容，可回寫為新頁或更新既有頁。
6. 在需要留下痕跡時，將本次查詢記錄到 `wiki/log.md`。

## 讀取順序

查詢時，預設依序讀：

1. `wiki/index.md`
2. 最相關的概念頁
3. 支撐這些概念頁的摘要頁
4. 必要時讀 `wiki/log.md` 最近紀錄，了解是否已有相近問題被整理過

除非 wiki 明顯不足，否則不要一開始就跳回 `raw/` 全量重讀。

查詢時不應因為看到日期就優先改走 `wiki/assets/daily/...` 或把回答改寫成閱讀回顧；除非任務本身已被判定為 `review`。

## 回答原則

1. 先用 wiki 回答，再判斷是否需要補充原始來源。
2. 優先引用概念頁中的整合結論，但重要主張要能追溯到摘要頁或來源。
3. 如果不同頁面有矛盾，不能硬湊成單一結論；應明確說明分歧。
4. 對時間敏感、版本敏感、條件敏感的資訊，要保留上下文。
5. 若 wiki 證據不足，要直接說明缺口，不要過度推論。

## 查詢策略

### 1. 事實型問題

先找最接近主題的概念頁，再向下驗證其引用的摘要頁。回答時簡潔，並附上依據頁面。

### 2. 比較型問題

同時讀多個概念頁與對應摘要頁，整理出：

- 共同點
- 差異點
- 條件差異
- 證據強弱
- 尚未解決的爭議

### 3. 綜合型問題

當問題跨多來源、多概念時，將回答視為一次知識編譯工作，而不是聊天即興輸出。必要時先列出將要依賴的頁面，再產出整合結論。

## 何時回寫 wiki

若本次查詢產出具持久價值的內容，應回寫 `wiki/`。例如：

- 一份高價值比較
- 一個跨多頁的新綜合結論
- 一個原本未獨立成頁但已明顯重要的概念
- 一個能改善後續查詢效率的總結頁

若回寫：

1. 優先更新既有概念頁。
2. 若需要新頁，放在適合的目錄並使用 kebab-case。
3. 更新 `wiki/index.md`。
4. 視情況在 `wiki/log.md` 記一筆 query 紀錄。
5. 若要更新 `wiki/log.md`，請依 `wiki/rules/log-rules.md`。

建議 query log 格式：

```md
## [YYYY-MM-DD] query | <question-or-topic>

- pages-read: <key pages>
- output: <answer page or summary>
- notes: <new synthesis, gaps, or follow-up questions>
```

## 缺口處理

若 wiki 不足以回答問題，應明確指出是哪一種缺口：

- 尚無對應摘要頁
- 有摘要但未整合成概念頁
- 多頁互相矛盾，尚未整理
- 缺乏更新來源

如果使用者要你補齊缺口，再轉入 ingest 或 lint 工作流。

## 輸出風格

1. 回答以清晰、可追溯為優先。
2. 優先使用 wiki 內部頁面作為依據，並且加上`source link` 代表外部原始來源網址，而不是空泛地說「根據資料」。
3. 若問題適合產出表格、清單、比較頁、簡報或其他結構化輸出，可採用，但內容仍應以 wiki 為依據。
4. 不要套用 review 報表欄位，例如「日期開場白」或「閱讀總覽」，除非使用者明確要看時間回顧。
5. 不可只回傳頁面標題、檔名、篇名或連結清單來取代分析；最終回覆必須包含整合後的結論或比較結果。

## 輸出規則

1. 最終輸出格式請依 `wiki/rules/output-rules.md`。

## 不要做的事

1. 不要跳過 `index.md`，直接盲讀整個 wiki。
2. 不要把暫時性的聊天輸出都留在對話中，卻不把高價值結論回寫。
3. 不要在證據不足時假裝 wiki 已有定論。
4. 不要忽略不同來源之間的時間差與版本差。
