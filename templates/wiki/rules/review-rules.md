# 知識庫回顧與報表規則 (Review Rules)

**目標：**  
使用者會透過通訊軟體詢問特定日期或時間區間的閱讀紀錄，例如「今天」、「昨天」、「4/18」、「4/18-4/20」、「這星期」、「這個月」。系統會提供當下日期作為上下文，你需要自行換算相對日期，例如「昨天」就是當天日期減一。你的任務是根據知識庫中已有的資料，整理出適合在手機上閱讀的精簡回顧。

## 執行步驟 (SOP)

### 1. 讀取資料

- 先判斷使用者詢問的是哪一種時間範圍：
  - **單日**：例如「今天」、「昨天」、「4/18」。
  - **短區間（<= 4 天）**：例如「4/18-4/20」、「最近三天」。
  - **長區間（> 4 天）**：例如「這星期」、「這個月」、「4/01-4/30」。
- 如果是 **單日** 或 **短區間（<= 4 天）**，請使用以下規則取得資料：
  - 先閱讀 `wiki/index.md`。
  - 從 `wiki/index.md` 的 `Summaries` 區塊確認該日期或區間內有哪些 summary 條目。
  - 依照 `## Summaries` 條目中的日期與路徑，打開對應的 `wiki/summaries/...` 頁面。
  - 這類情況應以 `wiki/summaries/...` 為主，因為需要較細的逐篇資訊。
- 如果是 **長區間（> 5 天）**，請使用以下規則取得資料：
  - 讀取時間區間內的 `wiki/assets/daily/<日期>.md`，檔名格式例如 `2026-05-21.md`。
  - 若區間內有多天，必須逐日打開對應的 daily 檔，不可只讀其中一天。
  - 這類情況應以 `wiki/assets/daily/...` 為主，因為 daily 檔已經是按日聚合後的回顧資料。
- 若使用者詢問的是長區間，但部分日期沒有對應的 `wiki/assets/daily/<日期>.md`，只能根據實際存在的 daily 檔整理，不可自行補齊不存在的日期內容。

### 2. 找不到資料時

- 如果系統提供的資料中沒有該日期的紀錄，請直接回覆：
  `您在這一天沒有新增任何閱讀紀錄喔！`
- 如果是時間區間查詢，但整個區間內都沒有任何紀錄，請直接回覆：
  `您在這段時間沒有新增任何閱讀紀錄喔！`
- 不要額外延伸。

### 3. 報表內容

```
- 日期開場白  
  例如：`📅 這是您在 使用者傳入的時間可能是範圍或是某天 的知識庫攝取紀錄：`
- 閱讀總覽  
  例如：`📚 共整理了 2 篇內容。`
- 逐篇摘要  
  每篇包含：
  - 主題名稱 例如：【Cloudflare Pull consumers】
  - `source link` 例如：source link : https://example.com/
  - 1 句話核心概念
  - 3 到 5 個重點
- 知識串聯（可選）  
  若同一天或同一區間內幾篇內容有明顯關聯，可補上一句整體觀察。
```

其中 `source link` 應優先代表外部原始來源網址；若同時需要本地素材位置，可額外附上 `source: raw/...`。

### 4. 限制

- 嚴格根據本次對話中提供的文字資料整理。
- 不可自行補充外部資訊。
- 不可憑空推論未提供的細節。
- 結尾不要主動發問，除非使用者原本有要求。

### 5. 輸出規則

- 最終輸出格式請依 `wiki/rules/output-rules.md`。

### 6. Log 規則

- 若這次回顧流程屬於實際執行的 Worker / 排程 / 系統任務，完成後必須同步更新 `wiki/log.md`。
- log 必須記錄這次回顧任務的時間範圍、使用了哪些關鍵頁面，以及任務輸出或結果。
- 若本次只是單純聊天查詢、沒有要求寫入知識庫檔案或執行系統流程，則不應為了查詢回覆額外改寫 `wiki/log.md`。

建議 review log 格式：

```md
## [YYYY-MM-DD] review | <date-range-or-question>

- pages-read: <key pages>
- output: <reply summary or generated artifact>
- notes: <time range, gaps, or follow-up>
```

## 補充說明

若 `index.md` 的 `Summaries` 已帶日期，則它是「單日 / 短區間」回顧的主入口，且該日期預設代表 `created`。
`wiki/assets/daily/...` 是「長區間」回顧的主入口，適合拿來做多日聚合，不應反過來取代單日查詢時的 summary 細節。
`wiki/concepts/...` 僅作為輔助理解，不是時間回顧的主資料來源。
若需要回覆原始來源，應優先使用摘要頁內的 `source link`；若摘要頁沒有，才回退到摘要頁內的 `source`。無論哪種情況，都不要反推、改寫或自行構造不存在的來源網址。
