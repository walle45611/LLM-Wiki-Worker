# Output Rules

本檔只定義輸出格式規則，不處理資料讀取、判斷流程、寫檔邏輯或 wiki 內容結構。

## 核心原則

同一次任務可以同時包含兩種輸出：

1. 寫入檔案的內容
2. 最後回覆給使用者的訊息

這兩種輸出應分開處理，不能混用格式。

## 1. 寫入檔案時

若本次任務需要修改或建立 `wiki/` 檔案，則：

1. 寫入檔案的內容應使用適合長期維護的 Markdown。
2. 可依任務需求使用標題、清單、表格、區塊與其他 Markdown 結構。
3. 檔案內容應重視可讀性、可追溯性與後續維護性。

## 2. 最後回覆給使用者時

若本次任務需要對使用者輸出最終訊息，則：

1. 最終回覆預設應使用 `zh-TW`。
2. 最終回覆禁止使用 Markdown。
3. 禁止輸出 Markdown 標題、粗體清單、表格、巢狀列表。
4. 禁止使用粗體語法，包括 `**文字**`、`__文字__`，以及任何用 Markdown 做強調的寫法。
5. 禁止輸出例如 `- **其他相關頁面**`、`- **重點整理**`、`## 標題`、`### 小節`、`1. 清單項目` 這類格式。
6. 需要分段時，使用純文字段落；若任務規則另有要求，可搭配 `【】` 標示。
7. 應適度加入表情符號，強化可讀性與掃讀性，但不可因此犧牲清楚度。
8. 若 task rule 提供的是欄位順序或段落結構，應保留該資訊架構，但改寫成純文字格式，不可因為禁用 Markdown 就省略欄位。
9. 可依任務需求使用短段落、空行、全形標點或 `【】` 欄位標記提升可讀性，但仍須維持純文字，不可退回 Markdown。
10. 不可為了追求精簡、手機可讀性或短訊息風格，就把 task rule 明確要求的欄位、省略成單純標題清單、連結清單或篇名羅列。

## 3. 同時有寫檔與回覆時

若同一次任務同時包含「修改檔案」與「回覆使用者」，則：

1. 寫入檔案的內容使用 Markdown。
2. 最後回覆給使用者的訊息使用純文字，且預設為 `zh-TW`。
3. 不可把要寫入檔案的 Markdown 內容整份直接回給使用者。

## 4. 套用方式

1. 本檔只管輸出格式，不取代其他 task rule。
2. 其他規則檔若提到輸出格式，應以本檔為準。
3. 若同一次任務同時觸發多個 rules，仍以本檔判斷「寫入檔案」與「回覆使用者」的輸出方式。
